What is Git?

Git is a version-control system (VCS) used to track changes made to projects. While Git seems simple, it's actually a bit challenging to get comfortable with. (Note that Git is a piece of software, on which version control sites like Github, Gitlab and others are based.)

Why is It Used?

Git was developed to allow multiple people to work on a project simultaneously without problems or complications. It was originally authored by Linus Torvalds, creator of Linux, as a way to manage all the contributors to the Linux kernel.

Git is an impressive bit of software, allowing distributed maintenance of a project repository with every computer with a git repository holding a full project history, even without a central server or network access.

Git has become incredibly common version-control system, employed on the popular developer sites Github and Gitlab. Despite it being so prevalent, Git can be intimidating to learn. We hope to make it a little easier! Let's start with a definition of common Git terms.

Common Git Terms

  • Repository A Git repository holds all the files for your Git project as well as a history of all changes made to that project, starting from its creation. It is the basic unit of Git: all other actions are defined by how they change or do not change the repository. It looks a lot like a normal project directory, except it has a folder titled .git. Repository is frequently shortened to "repo".
  • Commits In Git, commits signfies changes to a Git repository. It's similar to the "save" function on your computer. It takes a current snapshot of all the changes and creates a checkpoint. A Git commit includes:
    • Changes to the repository
    • When the changes were snapshotted
    • Who authored the changes
    • How the author describes the changes
    A commit is not used as frequently as you save your work locally. Instead, a commit is meant to represent a certain amount of work done on a project. In the commit message, you would then describe what that certain amount of work accomplished.
  • Remote Even though Git specializes in being a distributed version-control system, there is typically a central, common repository that all folks working on a project agree to keep updated with the changes, or commits, they're making to the repository on their machine. This central, common repository can be on a server at work or the public Github page of a project. In Git, we call this repository the remote repository.
  • Push / Pull The process of updating the central, or remote, repository with your commits is knowing as pushing your commits. Conversely, when you want to get the commits other folks have pushed to the remote repository, you pull those commits. Others on your project cannot see your changes until you've pushed your commit to remote. Pushing is for uploading your commits to the remote repository, pulling is for downloading others commits to the remote repository.
  • Branch A branch is an independent line of commits (changes) on a repository. Branches are usually created when you're conducting long-term development, such as building a major feature or upgrade or fixing a major bug.
  • Main is the default branch for development in a repository. It was previously called "master" but renamed in 2020.
  • Merge is combining of two separate branches of development. It joins together not only the files in the codebases, but also the commit history of the two separate branches.
  • Rolling Back a Commit Git allows you to develop without worries because you can't ever truly break code. If something breaks with a commit you've done, you can simply "roll back" the branch to before the offending commit. This is also known as resetting, reverting or returning to a previous commit.
  • Clone Cloning is the act of copying a desired repository to your computer. It's a quick way to download an entire project. Note that git clone will only copy the repository, you still have to cd into the repository's directory if you're on the command line.
  • Pull Request A pull request is when a contributor asks to merge a branch of development with another branch. This involves the maintainers of the second branch to review and comment on the contributor's code. Sometimes, a pull request is approved and the contributor's branch is merged into the other branch. A pull request typically contains the files to be changed if the two branches were to merge, description of what those changes are / accomplish, and a forum of some sort for the different parties to discuss the pull request. A pull request is not a part of the core Git software, but rather is a feature of Git services like Github or Gitlab.
